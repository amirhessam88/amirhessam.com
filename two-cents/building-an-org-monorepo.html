🏗️ Building an Org Monorepo

📅 July 23, 2026 · ⏱️ 28 min read

📖 Inspiration. In 2021, I first read Dan Hipschman’s Our Python Monorepo (Opendoor / Open House) [1] when I was designing this approach. That write-up clarified the polyrepo failure modes — painful library bumps, untestable shared code, copy-paste sprawl, and divergent lint configs — and showed that a single repo with path-linked libraries + shared CI templates fixes the graph. What follows is our pattern: same core invariants, tightened into a stricter topology (apps/, libs/, projects/, services/, dockers/, cookiecutters/, iac/, docs/), org-namespaced packages, GitHub xor GitLab, and a modern uv + poethepoet toolchain.

💡 An org monorepo is not “one giant Git tree.” It is a versioned dependency graph with enforceable import edges, a single toolchain contract, reusable CI/CD templates, and IaC that ships next to the code it provisions. Below is the blueprint we use.

🌐 Language note. Most concrete recommendations below are Python-flavored (uv, ruff, mypy/pyright, poe, tox). The pattern itself is language-agnostic: the same topology (apps/, libs/, projects/, services/, dockers/, cookiecutters/, iac/, docs/), import/dependency DAG rules, leaf CI child nodes, org-namespaced libraries, scaffolder CLIs, and docs portal aggregation (e.g. MkDocs) work for Python, Go, Rust, and other stacks in the same monorepo — swap the per-language toolchain, keep the contracts.

😵‍💫 Why polyrepos rot under shared code

Most platform teams rediscover the same failure modes once shared Python code shows up across services:

  • 🔁 Library updates require sequential PRs across producer + consumer repos, plus an internal package index bump — merge the lib, wait for the wheel to publish, then bump the consumer.
  • 🧪 You cannot cheaply prove a lib change is safe against the full dependent closure; nobody runs “every service’s tests” before merging.
  • 📋 Engineers stop extracting libraries and start copy-pasting, because sharing is too expensive.
  • 🌀 Accidental cycles and fuzzy ownership appear the moment “shared helpers” leak between services inside a junk-drawer repo.
  • 🔎 Cross-repo search and IDE navigation are worse than a single workspace checkout.
  • 🧭 Per-repo lint/format/type configs diverge until cross-repo contribution becomes a tax.

Our monorepo collapses publish-then-consume into path / editable installs via uv workspaces [2]. CI expands the affected closure from the workspace graph / uv.lock. The invariant: internal packages are linked, not version-chased. External consumers outside the monorepo can still get versioned wheels from an internal index if needed — but inside the repo, path deps are law.

🎙️ Personal note. I personally have some issues with Charlie Marsh’s (Astral founder) mindset (as discussed here) — but uv is still a great product, and I’d standardize an org monorepo on it anyway.

📁 Target topology

Keep top-level directories semantic and mutually exclusive. Every leaf package owns a pyproject.toml and a CI entrypoint: .gitlab-ci.yml on GitLab, or the GitHub equivalent (a thin leaf workflow / config that uses: a reusable workflow — details below). Nothing cross-cuts via relative imports into another leaf’s src/.

🔐 VCS exclusivity: an org standardizes on either GitHub or GitLab — never both as first-class homes for the same monorepo. Keep exactly one of .github/ or .gitlab/ for shared templates, child-pipeline / reusable-workflow entrypoints, and ownership metadata.

org-monorepo/
├── .github/                 # XOR: workflows/, actions/, CODEOWNERS
│   # — or —
├── .gitlab/                 # XOR: ci/ templates, includes, CODEOWNERS
├── apps/                    # user-facing / operator-facing applications
├── libs/                    # shared abstractions (org-namespaced packages)
├── projects/                # domain-bounded workspaces (ETL, ML, batch, research)
├── services/                # independently deployable microservices
├── dockers/                 # shared Dockerfiles, base images, compose stacks
├── cookiecutters/           # templates for poe new-* scaffolds (apps, libs, …)
├── iac/                     # Terraform / Pulumi / cloud foundation modules
├── docs/                    # docs portal / landing — aggregates leaf docs via CD
├── pyproject.toml           # uv workspace root + shared tool config
├── uv.lock                  # deterministic resolver output
└── README.md

🔎 Directory contracts

  • 🚀 apps/ — deployable application binaries / UIs / CLIs with a release artifact. May depend on libs/*. Must not import services/* or sibling apps.
  • 📦 libs/ — the only approved sharing layer. Logging, auth clients, protobuf stubs, feature flags, schema helpers, ML feature transforms — anything with a stable public API and tests. Naming is org-namespaced (see below).
  • 🧬 projects/ — domain-centric workspaces (e.g. projects/risk-scoring, projects/claims-etl). These are not microservices; they are bounded contexts that may produce jobs, notebooks (exported), or training pipelines. Depend on libs/*, never on apps/*.
  • 🛰️ services/ — network-facing microservices (HTTP/gRPC/workers) with their own image + deploy pipeline. Import only from libs/*. Prefer building FROM shared bases in dockers/ rather than copy-pasting Dockerfiles.
  • 🐳 dockers/ — shared container assets: org base images, common Dockerfiles, multi-stage templates, and local compose stacks (deps, mesh stubs). Leaves reference these; they do not reinvent FROM python:… in every app/service. Image build/push CD still runs from the leaf that owns the final artifact, using dockers/ as build context / FROM source.
  • 🍪 cookiecutters/ — golden-path templates for new leaves (python-lib, python-service, js-app, …). Downstream engineers scaffold via task-runner CLIs (poe new-python-lib, poe new-js-app) — they should not need to learn the nuts and bolts of workspace wiring, CI stubs, or naming maps. See the Cookiecutters section.
  • ☁️ iac/ — Terraform (or equivalent) modules and live stacks. App/service code does not import IaC; CI wires plan/apply via path filters.
  • 📚 docs/ — the docs portal landing page and shared chrome (nav, style, ADRs, org-wide guides). Downstream leaves own their own docs/ nodes; CD builds them on the fly and publishes one site. See the Docs section.
  • 🧩 .github/ or .gitlab/ — shared templates + the parent pipelines that fan out into leaf jobs (child pipelines / reusable workflows). See CI section.

🧾 What every leaf owns

A leaf is not “just source.” It ships its package manifest and its CI contract:

# GitLab leaf
libs/foo-bar-baz-qux/
├── pyproject.toml
├── .gitlab-ci.yml           # child-pipeline entry (thin; includes shared template)
├── docs/                    # leaf docs node (API + how-to) — aggregated by CD
├── src/foo/bar/baz/qux/
└── tests/

# GitHub leaf — Actions workflows must live under repo-root .github/workflows/
# so the leaf owns a thin config; the stub workflow lives next to shared templates
libs/foo-bar-baz-qux/
├── pyproject.toml
├── ci.yml                   # leaf values: image, matrix, deploy target, …
├── docs/                    # leaf docs node — aggregated by CD
├── src/foo/bar/baz/qux/
└── tests/

.github/workflows/
├── _python-leaf.yml         # reusable workflow (workflow_call) — the real jobs
└── libs-foo-bar-baz-qux.yml # stub: path filters + uses: _python-leaf.yml

GitHub’s equivalent of a per-directory .gitlab-ci.yml is not a nested workflow file inside libs/… (Actions only loads workflows from .github/workflows/ at the repo root). The practical equivalent is: leaf ci.yml (values) + root stub workflow that uses: a reusable workflow. Same idea as GitLab — leaf declares intent; shared template owns mechanics.

🏷️ Lib naming: hyphenated distro → nested import path

Library distribution names are org-based namespaces written with hyphens. Hyphens map 1:1 to nested packages on disk and in site-packages after install:

# distribution / directory name          →  import / site-packages path
foo-bar-baz-qux                          →  foo/bar/baz/qux
acme-ml-features-transforms              →  acme/ml/features/transforms
acme-platform-logging                    →  acme/platform/logging

Source layout inside the leaf (CI files shown above):

libs/foo-bar-baz-qux/
├── pyproject.toml          # name = "foo-bar-baz-qux"
├── src/
│   └── foo/
│       └── bar/
│           └── baz/
│               └── qux/
│                   ├── __init__.py
│                   └── ...
└── tests/

After uv sync / pip install, the package lands as nested modules under site-packages:

site-packages/
└── foo/
    └── bar/
        └── baz/
            └── qux/
                ├── __init__.py
                └── ...

# consumers import the namespace path, not the hyphenated distro name
from foo.bar.baz.qux import SomeType

Why this matters:

  • 🧭 Collision resistance — org prefixes (acme-…, foo-…) keep internal libs from clobbering top-level PyPI names in site-packages.
  • 📂 Discoverability — related libs cluster under the same tree (acme/ml/…, acme/platform/…) in both the repo and the installed environment.
  • 🔗 Stable mental model — libs/<distro> ↔ src/<slash-path> ↔ from <dot-path> import … is mechanical: replace - with / (or . for imports).

Hard rules: distribution name is always hyphenated kebab-case; Python packages never use hyphens; do not publish a flat foobarbazqux module that ignores the namespace map.

🧱 Import DAG rules (non-negotiable)

  • ✅ apps|projects|services → libs
  • ✅ libs → libs only via an acyclic internal graph (enforce with import-linter / custom ruff plugin / CI graph check).
  • 🚫 libs → apps|projects|services
  • 🚫 apps ↔ services (talk over APIs / events, not Python imports)
  • 🚫 deep imports into another package’s private modules — public entry points only

Hard organizational rule: apps/*, projects/*, and services/* never import each other. If two leaves need the same code, it graduates into libs/. Encode that in tooling (import-linter / CI graph check), not wiki pages.

🧰 Base toolchain (Python)

Standardize the developer loop on a fast resolver, one task runner, strict static analysis, and multi-env tox. Every tool below should be pinned in the workspace root and invoked through poe so humans and CI share identical entrypoints.

⚡ uv [2]

Package/project manager + workspace linker. Use uv sync --locked at the root, path deps for libs/*, and uv.lock as the source of truth for affected-package computation. Constraint solving + lockfiles keep local, CI, and prod on the same graph — with materially better cold-install performance than older Poetry-era stacks.

🧹 Ruff [3]

Single-binary lint + format. Configure once at the workspace root ([tool.ruff]); leaf packages may only tighten, never invent incompatible dialects. Wire it as poe format / ruff check inside poe check.

🧠 Type checking — mypy [4] or Pyright [5]

Pick one org-wide and enforce disallow_untyped_defs (mypy) or typeCheckingMode = "strict" (Pyright) on libs/* first. Apps and services can ratchet up afterward. Expose the chosen checker as poe mypy (or poe pyright); Pyright is the better fit if you want language-server parity with VS Code / Pylance.

🎭 Poe the Poet [6]

Task runner declared in pyproject.toml. This is the control plane: format, check, test, tox, build, codegen, etc. Engineers learn poe check && poe test; CI calls the same tasks via [tool.poe.tasks].

🧪 tox + tox-uv [7] [8]

Matrix execution across Python versions / factor envs without hand-rolled shell. With tox-uv, envs resolve through uv so local and CI stay aligned. Expose as poe tox.

# root pyproject.toml — illustrative shape
[tool.uv.workspace]
members = ["apps/*", "libs/*", "projects/*", "services/*"]

[tool.poe.tasks.format]
cmd = "ruff format ."

[tool.poe.tasks.check]
sequence = [
  { cmd = "poe format --check" },
  { cmd = "ruff check ." },
  { cmd = "poe mypy" },
]

[tool.poe.tasks.test]
cmd = "pytest -q"

[tool.poe.tasks.tox]
cmd = "tox"

🧩 CI/CD — parent templates, child nodes where jobs start

Pick one forge and commit to it. The root .github/ or .gitlab/ tree is the parent: shared job definitions, images, rules, and deploy primitives. Leaf CI files are child nodes — thin entrypoints that start the real jobs by including / calling those templates. Do not copy-paste a full pipeline into every leaf.

🦊 GitLab — .gitlab/ + per-leaf .gitlab-ci.yml

.gitlab/
├── ci/
│   ├── python-leaf.yml      # shared jobs: lint, typecheck, test, build
│   ├── docker-deploy.yml    # shared deploy jobs for apps/services
│   └── iac-plan-apply.yml
└── CODEOWNERS

# root .gitlab-ci.yml — parent pipeline / dispatcher
include:
  - local: .gitlab/ci/python-leaf.yml   # optional defaults
stages: [detect, test, build, deploy]

# fan-out: trigger a child pipeline per affected leaf
trigger-libs-foo-bar-baz-qux:
  stage: test
  trigger:
    include:
      - local: libs/foo-bar-baz-qux/.gitlab-ci.yml
    strategy: depend
  rules:
    - changes:
        - libs/foo-bar-baz-qux/**/*
        - libs/foo-bar-*/**/*          # or compute dependents from uv.lock

Each leaf’s .gitlab-ci.yml stays tiny — it includes .gitlab/ci/python-leaf.yml and only overrides variables (image, pytest path, whether to publish a wheel). Jobs start in the child pipeline for that leaf, not as a 500-line monolith at the repo root.

🐙 GitHub — .github/workflows/ reusable workflows

GitHub Actions does not discover workflow YAML inside apps/ or libs/. Workflows live under .github/workflows/. The equivalent of GitLab child pipelines is reusable workflows (on: workflow_call) plus per-leaf stub workflows (or one dispatcher workflow) that pass leaf inputs:

.github/
├── workflows/
│   ├── _python-leaf.yml           # reusable: the jobs (lint/test/build)
│   ├── _docker-deploy.yml         # reusable: build/push/deploy
│   ├── libs-foo-bar-baz-qux.yml   # child node / stub — where this leaf starts
│   └── monorepo-dispatch.yml      # optional: path-filter → matrix of stubs
├── actions/                       # optional composite actions
└── CODEOWNERS

# libs-foo-bar-baz-qux.yml (stub — child node)
name: libs/foo-bar-baz-qux
on:
  pull_request:
    paths: ["libs/foo-bar-baz-qux/**", "libs/**/pyproject.toml"]
  push:
    paths: ["libs/foo-bar-baz-qux/**"]
jobs:
  python:
    uses: ./.github/workflows/_python-leaf.yml
    with:
      working-directory: libs/foo-bar-baz-qux
      # or read libs/foo-bar-baz-qux/ci.yml inside the reusable workflow

So: reusable workflow = shared job body; stub workflow (or dispatcher matrix entry) = child node where that leaf’s jobs start. Leaf-local ci.yml holds values only — image, matrix, deploy target, Slack channel, “needs DB?”, etc.

Always parameterize the same four concerns:

  • 🎯 path filters → which leaf’s child node wakes up
  • 🕸️ workspace graph → which dependents must re-test
  • 🐳 image build + push for apps/* / services/* on leaf changes; rebuild shared bases under dockers/ when those Dockerfiles change, then rebuild dependents
  • ☁️ iac/* → terraform plan on PR, apply on protected branches

💓 Repo-Health — if you don’t measure it, you can’t improve it

A monorepo without measurable quality is just a bigger place to hide regressions. Unit tests, integration tests, version/arch matrices, and repo-wide coverage are how you prove that a change to libs/* is safe for every downstream apps/, projects/, and services/ consumer — not hope.

🧪 Unit + integration tests with pytest

Standardize on pytest [29] in every Python leaf:

  • 🔬 Unit tests — fast, no network, live next to the package (tests/unit/). Required on every PR child pipeline.
  • 🔗 Integration tests — tests/integration/ against realistic boundaries (Testcontainers, compose from dockers/compose/, LocalStack, ephemeral DBs). Mark with @pytest.mark.integration; run on PR for touched leaves, and more broadly on a schedule.
  • 📦 Libs especially — a library’s unit suite is necessary but not sufficient. Prefer contract / consumer-style tests that exercise the public API the way apps/* and services/* will call it. 
# every Python leaf
libs/foo-bar-baz-qux/
├── src/foo/bar/baz/qux/
├── tests/
│   ├── unit/
│   └── integration/
└── pyproject.toml          # pytest + pytest-cov config

🐍 tox matrices — guarantee downstream runtimes

Downstream leaves will not all pin the same Python patch or CPU arch. Use tox (+ tox-uv) so each libs/* (and critical services) proves itself across the matrix your org actually runs:

# libs/foo-bar-baz-qux/tox.ini  (illustrative)
[tox]
env_list = py310, py311, py312, py313

[testenv]
runner = uv-venv-lock-runner
commands =
    pytest tests/unit -q --cov=foo.bar.baz.qux --cov-report=xml

# optional factors — OS / arch via CI matrix, not only tox envs
# e.g. GitHub: strategy.matrix = { python: [...], os: [ubuntu, macos] }
  • 🧩 PR path — affected leaf runs poe test (current default Python) + a lean tox subset if the leaf is a shared lib.
  • 🌐 Full matrix — poe tox on shared libs/* for every supported CPython, and CI runners for arch/OS (amd64 / arm64, linux / macos) when you ship wheels or multi-arch images.
  • 🛡️ Why this matters — when services/billing is on 3.11-arm64 and apps/console is on 3.12-amd64, a green single-env lib PR is a lie. The matrix is how you make downstream use guaranteed, not assumed.

🌙 Nightly: crawl the monorepo, accumulate coverage

PR CI proves the diff. Nightly proves the estate. Schedule a parent pipeline (GitHub schedule / GitLab pipeline schedule) that:

  1. 🕷️ Crawls every testable leaf under apps/*, libs/*, projects/*, services/* (discover pyproject.toml / tox.ini / package.json, etc.).
  2. 🧪 Runs unit (+ optional integration) suites per leaf — Python via poe test / tox, other languages via their leaf task.
  3. 📊 Emits per-leaf coverage artifacts (coverage.xml, lcov, …).
  4. 🧮 Accumulates into a monorepo rollup (e.g. merge coverage.py data with coverage combine, or upload all partials to Codecov/Coveralls/Sonar with a monorepo flag).
  5. 📈 Publishes a trend: total %, per-top-level-dir %, worst leaves, and a diff vs. last night — fail the nightly (alert; don’t block product merges) if coverage drops beyond a threshold.
# conceptual nightly (forge-agnostic)
discover leaves
for leaf in leaves:
    (cd $leaf && poe test --cov)   # or poe tox on libs
upload partial coverage artifacts
coverage combine ./coverage-partials/*   # or vendor uploader
publish report → docs portal / coverage dashboard
route anomalies → CODEOWNERS for failing leaves

Reference stack: pytest + coverage.py [30] locally; Codecov / Coveralls / Sonar (or a simple HTML report published next to the docs portal) for the accumulated nightly view. Same rule as everything else: one poe coverage-nightly (or equivalent) that CI calls — no snowflake nightly scripts per leaf.

🚨 Alerting CODEOWNERS on nightly anomalies

When the nightly crawler finds something wrong, do not spam #engineering. Route to the CODEOWNERS of the failing leaf (and platform only for rollup / infra symptoms). Ground the policy in Rob Ewaschuk’s My Philosophy on Alerting [31] (widely cited in Google SRE practice): pages should be urgent, important, actionable, and real — prefer symptoms over raw causes, and keep cause detail on dashboards / in the alert body for debugging.

Map that to Repo-Health nightlies:

  • 🩺 Alert on symptoms — e.g. “leaf libs/foo-bar-baz-qux nightly is red,” “monorepo coverage dropped > N% WoW,” “tox py312 factor failed for 3 shared libs,” “integration suite timed out for services/billing.” These are user/platform pain signals for that path’s owners.
  • 🔍 Attach causes in the payload / dashboard — failed test names, flake retries, Python version, runner arch, coverage Δ, last green SHA — so owners can debug without a second page for every proximate cause.
  • 👤 Route by CODEOWNERS — resolve /libs/foo-bar-baz-qux/ → @org/foo-bar-owners (Slack/Teams/PagerDuty/ GitHub Issues). Platform owns rollup symptoms (dockers/, CI templates, crawler itself).
  • 🔇 Don’t page on every cause — a single flaky test failure is a ticket/dashboard item; a leaf that was green for 14 days and is now red is a page. Err toward removing noisy alerts.

Example anomaly classes the nightly should detect and fan out:

  • 💥 Suite regression — previously green leaf now failing unit / integration / tox env
  • 📉 Coverage cliff — leaf or monorepo total drops beyond threshold
  • 🐍 Matrix break — lib fails on a Python/OS/arch still required by downstream apps/services
  • ⏱️ Health of the crawler — nightly itself didn’t finish / couldn’t discover leaves (symptom: “Repo-Health blind” → page platform)
  • 🔁 Flake storms — same test fails intermittently above a rate (ticket the owners; page only if a release path is blocked)
# conceptual alert routing
for anomaly in nightly_anomalies:
    paths = anomaly.affected_paths          # e.g. libs/foo-bar-baz-qux/**
    owners = resolve_codeowners(paths)      # from CODEOWNERS
    notify(
        owners,
        symptom=anomaly.symptom,            # "nightly red" / "coverage cliff"
        causes=anomaly.debug_bundle,        # logs, SHA, matrix cell, …
        runbook=docs_url("repo-health"),
    )

Observability here is not just dashboards — it is closed-loop ownership: measure the estate, detect symptoms, page the people who can fix that path, and keep causes visible without turning every log line into a pager event [31].

🔥 Blaze / Bazel and Buck — why we did not start there

Two hyperscale lineages dominate this design space:

  • 🟠 Google — Blaze → Bazel [13] — hermetic actions, fine-grained target graph, remote caching / remote execution, Starlark rules. Bazel is the open-source Blaze.
  • 🔵 Meta — Buck → Buck2 [22] — Meta’s monorepo build system (Buck1 open-sourced earlier; Buck2 is the from-scratch Rust rewrite). Same class of problem: polyglot targets, hermeticity, remote execution (REAPI-compatible), rules in Starlark decoupled from the core [23].

For a hyperscale polyglot monorepo, Bazel or Buck2 is often the correct endgame. Both are the wrong first move for most orgs that just need shared libs + services in one Git repo.

✅ Pros (Bazel / Buck2 class)

  • 🧮 Hermetic, reproducible builds — inputs hash to outputs; “works on my machine” shrinks dramatically.
  • ⚡ Incremental + remote cache — unchanged targets are fetched, not rebuilt; CI and laptops can share a cache.
  • 🕸️ Precise affected graph — test/build only what the target graph says changed, across languages.
  • 🌍 Polyglot-native — one system for Python, Go, Rust, Java/C++, protos, container images, etc.
  • 🔁 Buck2-specific upside — dynamic dependencies (graph can grow mid-build from tool output), rules fully outside the Rust core, strong parallelism story vs Buck1 / vs phase-heavy designs.

❌ Cons (same tax either way)

  • 🧗 Steep learning curve — BUILD/BUCK files, Starlark rules, cells/workspaces, and rule ecosystems become a specialization.
  • 🧰 Toolchain takeover — everyday language UX (uv, pytest, cargo test, IDE runners) often needs wrappers; onboarding tax is real.
  • 👥 Platform bottleneck — one or two engineers become the only people who can land build changes.
  • 💸 Ops cost — remote cache / RBE, rules upgrades, and language-rules churn are ongoing platform work, not a weekend migration.
  • 🥜 Overkill early — for dozens of leaves, Blaze/Buck-class hermeticity is a sledgehammer.
  • 🧪 Ecosystem maturity tradeoffs — Bazel has the broader open-source rules surface today; Buck2 is battle-tested at Meta but the public ecosystem / release cadence is younger — picking either is a multi-year platform bet.

🚫 Why we did not use Bazel or Buck first

We wanted a monorepo that engineers can extend on day two without learning a second build language. The uv workspace + poe tasks + forge-native child pipelines give us path deps, affected CI, and shared templates with tools people already know. Blaze/Bazel and Buck/Buck2 stay on the roadmap for when polyglot scale or CI minutes force hermetic remote execution — not as the admission price to have libs/ and services/ in one Git repo.

Rule of thumb: earn Bazel or Buck2. Start with the graph + CI contracts in this post; graduate to hermetic monorepo builders when the complexity tax is clearly cheaper than the CI bill — then choose based on your language mix, rules ecosystem, and whether you already have REAPI infrastructure.

🐳 Shared container assets — dockers/

Centralize image plumbing so every leaf does not ship a private snowflake Dockerfile:

dockers/
├── bases/
│   ├── python-runtime/Dockerfile    # org Python base
│   └── go-runtime/Dockerfile
├── templates/
│   └── python-service/Dockerfile    # multi-stage template leaves COPY from
├── compose/
│   ├── local-deps.yml               # postgres, redis, localstack, …
│   └── mesh-stub.yml
└── README.md

Leaves keep a thin Dockerfile (or build args) that FROM the org base / template. CD rebuilds dockers/bases/* on change, then rebuilds dependent apps/* / services/* images.

🍪 Cookiecutters — scaffold, don’t teach the bolts

The monorepo’s internal contracts (workspaces, CI stubs, naming maps, docs nodes, CODEOWNERS rows) are platform knowledge. Downstream users should not need to learn them. Keep golden-path generators under cookiecutters/ (name is historical — the engine can be any of the tools below) and expose them only through task-runner entrypoints (poe new-*).

🛠️ Scaffolding engines — pick one org-wide

  • 🍪 Cookiecutter [26] — classic Jinja templates; fire-and-forget generation. Simple, ubiquitous, polyglot. Weak at updating already-scaffolded leaves when the template evolves.
  • 📋 Copier [27] — Cookiecutter’s smarter cousin: Jinja templates + answers file + template updates with 3-way merges. Strong default when you want a golden path that can evolve without rewriting every leaf by hand.
  • 🧬 Projen [28] — project config as code (synthesize package.json, CI, lint, … from a typed .projenrc). Generated files are managed, not hand-edited — best when you want “cattle, not pets” consistency and a central push of config upgrades (popular in TS/CDK worlds; works beyond JS via jsii).
  • 📦 Other options in the same niche — Yeoman, Crane, language CLIs (cargo new, npm create) wrapped by poe so the user surface stays poe new-* regardless of engine.

Recommendation: Copier for polyglot monorepos that need template upgrades; Cookiecutter if you only need create-once scaffolding; Projen when most leaves are TS/Node (or CDK) and you want synthesized, non-editable project config. Do not mix three engines without a very good reason — the directory can still be called cookiecutters/ even if the binary is copier.

cookiecutters/
├── python-lib/               # Copier/Cookiecutter template — or
├── python-service/           #   projen project type definitions
├── python-project/
├── js-app/
├── go-service/
└── README.md                 # which poe task → which template/engine

# root pyproject.toml — illustrative (swap cookiecutter ↔ copier as needed)
[tool.poe.tasks.new-python-lib]
help = "Scaffold a namespaced Python lib under libs/"
cmd = "copier copy cookiecutters/python-lib libs"

[tool.poe.tasks.new-python-service]
cmd = "copier copy cookiecutters/python-service services"

[tool.poe.tasks.new-js-app]
cmd = "copier copy cookiecutters/js-app apps"

# if using projen instead:
# [tool.poe.tasks.new-js-app]
# cmd = "projen new --from ./cookiecutters/js-app …"

Each template / project type should emit a complete leaf in one shot:

  • 📦 correct directory + org-namespaced package layout
  • 🧾 pyproject.toml / language manifest wired into the workspace
  • 🧩 CI child stub (.gitlab-ci.yml or GitHub stub + ci.yml)
  • 📚 starter docs/ node
  • 👤 CODEOWNERS snippet / team placeholder
  • 🐳 thin Dockerfile that uses FROM with dockers/ bases when relevant

UX goal: poe new-python-lib → answer three prompts → open a PR. No wiki spelunking required. Platform owns the templates; product teams consume the CLIs.

🛡️ CODEOWNERS, ACL, and merge gates

A monorepo without path-level ownership is a shared writable disk. Every downstream leaf needs an explicit owner set, and the forge must refuse to ship without their approval.

# .github/CODEOWNERS  (or .gitlab/CODEOWNERS)
# Default — platform team
*                                @org/platform

# Leaf ownership — required reviewers for that path
/libs/foo-bar-baz-qux/           @org/ml-platform @org/foo-bar-owners
/services/billing/               @org/billing
/apps/console/                   @org/console
/dockers/                        @org/platform
/cookiecutters/                  @org/platform
/iac/                            @org/platform @org/sre
/docs/                           @org/platform
  • 🔒 Branch protection / MR approval rules — require CODEOWNERS review on the default branch; dismiss stale approvals on new pushes; block merge if checks fail.
  • 👥 Leaf teams own leaves — product groups are listed on their paths; platform owns dockers/, cookiecutters/, CI templates, and shared libs/* that many consumers depend on (or a dedicated libs review group).
  • 🚫 No self-merge of critical paths — especially iac/, deploy templates, and base images.
  • 🪪 ACL beyond Git — registry push, cloud deploy roles, and secrets access mapped to the same owner teams (OIDC / environment protection rules on GitHub; protected environments on GitLab). CODEOWNERS stops bad merges; cloud ACL stops bad deploys.
  • 📜 Document the contract in docs/ / CONTRIBUTING.md — who owns what, how to request ownership changes, and that unapproved path changes do not ship.

🤖 How AI tools help maintain the repo

Treat AI the same way you treat CI: encoded policy, not vibes. The monorepo should teach agents (and humans) the contracts so scaffolds, refactors, docs, and reviews stay on the golden path. Below is a practical operating model.

📐 1. Encode the contracts where agents look first

  • 📁 .cursor/rules/ — short Cursor rules for topology, naming (foo-bar → foo/bar), import DAG bans, “use poe new-* — don’t hand-roll leaves,” docs-node requirements, and PR style.
  • 📝 CLAUDE.md / Claude project rules — same invariants for Claude Code / Claude in the IDE so you are not maintaining two contradictory policy worlds.
  • 📖 docs/ + CONTRIBUTING.md + ADRs — source of truth; rules should link here, not duplicate novels.
  • 🧪 docs/ai/ — versioned prompts and checklists (pr-review.md, scaffold-leaf.md) that both humans and bots run.

🛠️ 2. Day-to-day maintainer jobs AI is good at

  • 🍪 Scaffold leaves — “add libs/acme-billing-client” → agent runs poe new-python-lib (Copier/Cookiecutter/Projen), fills prompts, opens a branch. No free-form file inventing.
  • 📚 Keep docs honest — when a public API changes, agent updates the leaf docs/ node and flags portal nav gaps before merge.
  • 🔁 Propagate template upgrades — after cookiecutters/ or CI templates change, agent opens MRs that re-sync leaves (especially strong with Copier update / Projen synth).
  • 🧹 Repo hygiene — find import-DAG violations, missing CODEOWNERS rows, orphaned Dockerfiles, stale path filters, docs-only leaves without poe docs coverage.
  • 🧾 Changelog / migration notes — draft ADR stubs and contributor-facing migration steps when a shared lib breaks consumers.
  • 🐛 CI archaeology — read failing job logs via MCP, propose the minimal fix, re-run the leaf child pipeline.

🔌 3. MCP — give agents tools, not pastebins

  • 🧰 Wire MCP servers for the forge (GitHub/GitLab), issue tracker, docs search, and package registry metadata so agents can open MRs, inspect checks, and query ownership without brittle shell one-offs.
  • 🔐 Scope credentials: read-heavy for review bots; write only for a bot identity that obeys the same ACL as humans.
  • 🧭 Prefer MCP + in-repo rules over stuffing the entire monorepo into a single prompt context window.

🕵️ 4. Review agents on every PR / MR

  • 🤖 On pull_request / merge_request open/sync, run a review agent (Actions / GitLab job, or hosted bots such as Copilot / CodeRabbit-class reviewers) against docs/ai/pr-review.md.
  • Checklist the agent should enforce:
    • import DAG + no leaf→leaf Python imports
    • org namespace naming on new libs
    • CI stub + docs/ node present for new leaves
    • no secrets; Dockerfiles FROM org bases
    • CODEOWNERS path touched has owners listed
  • ✅ Make the result a required check (pass / fail) or at least a blocking “changes requested” comment style — not a silent Slack side-channel.
  • 👤 Agents never bypass CODEOWNERS — humans still approve; the bot only fails the check or leaves review comments.

⚠️ 5. Guardrails — what AI must not do

  • 🚫 Invent a new top-level directory or skip poe new-* for “speed.”
  • 🚫 Self-approve or merge using admin tokens.
  • 🚫 Edit iac/, deploy templates, or dockers/bases without the owning team in the loop.
  • 🚫 Treat generated Projen/Copier-managed files as free-edit surfaces when the engine owns them.

Bottom line: AI maintains the monorepo by executing and enforcing written contracts — Cursor/Claude rules for authors, MCP for tool access, PR/MR agents for review — while ACL + CODEOWNERS remain the hard gate that prevents unapproved changes from shipping.

☁️ IaC next to the graph

Keep Terraform (or Pulumi) under iac/ with the usual module vs. stack split:

iac/
├── modules/          # reusable primitives (network, k8s, bucket, db)
├── stacks/
│   ├── platform/     # shared cluster / mesh / observability
│   └── services/     # per-service wiring (IAM, DNS, secrets refs)
└── README.md

Hard rule: 🔐 no secrets in Git — remote state + vault / cloud secret manager only. Service deploy pipelines consume outputs; they do not embed credentials.

Reference: Terraform docs [9].

📚 Docs portal — landing page + aggregate from leaves

Top-level docs/ is not a junk drawer of markdown. It is the documentation portal: the landing page, global nav, org ADRs, onboarding, and the glue that mounts every downstream leaf’s docs into one published site.

docs/                              # portal root (landing + chrome)
├── index.md                       # landing page
├── getting-started.md
├── adr/                           # architecture decision records
├── architecture/                  # C4 / Mermaid / D2
├── mkdocs.yml                     # or conf.py — portal config
└── overrides/                     # theme chrome

libs/foo-bar-baz-qux/docs/         # downstream docs node
├── index.md
└── api/                           # or rely on autodoc / mkdocstrings

services/billing/docs/
├── index.md
├── runbooks/
└── diagrams/

projects/claims-etl/docs/
└── index.md

🚢 CD builds docs on the fly and publishes

Do not ask humans to copy leaf markdown into docs/. A docs CD child node (same parent/child pattern as CI) should:

  1. 🔍 Discover docs nodes under apps/*/docs, libs/*/docs, projects/*/docs, services/*/docs.
  2. 🧩 Generate / assemble the portal nav (mount each leaf under e.g. /libs/foo-bar-baz-qux/, /services/billing/).
  3. ⚙️ Build API reference from docstrings where configured (autodoc / mkdocstrings).
  4. 📦 Emit a static site artifact.
  5. 🚀 Publish on merge to the default branch (GitHub Pages, GitLab Pages, S3+CloudFront, or Read the Docs).

Path filters: a leaf-only docs change rebuilds that leaf’s section (or a cheap full static rebuild — usually fine). A portal chrome change under top-level docs/ rebuilds the whole site. Wire this as poe docs locally and the same task in the docs CD child.

🐍 Python docs tooling — options

Pick one stack org-wide. Mixing Sphinx in half the leaves and MkDocs in the other half is how portals die.

  • 📘 MkDocs + Material for MkDocs + mkdocstrings [14] [15] [16] — markdown-first portal, excellent UX, easy multi-section nav. mkdocstrings pulls Python API docs from docstrings. Best default for a monorepo landing page that aggregates many leaves.
  • 📗 Sphinx + autodoc + MyST + Furo [17] [18] [24] — deepest Python API / cross-ref story; steeper config. Prefer when you need rich intersphinx, complex API trees, or already live in Sphinx. Live example: docs.slickml.com [25] (Sphinx + Furo).
  • 📄 pdoc [19] — low-ceremony API sites from docstrings. Great per-lib artifact; weaker as the org-wide portal unless you wrap it.
  • 📓 Jupyter Book [20] — best when projects/* are notebook-heavy; usually a section inside the portal, not the whole portal.
  • ☁️ Hosting — GitHub Pages / GitLab Pages for forge-native CD; Read the Docs [21] if you want versioned docs / PR previews without running your own Pages bucket.

Recommendation: MkDocs Material at docs/ as the portal, mkdocstrings pointed at each libs/* package path, leaf narrative docs living under each leaf’s docs/, assembled by poe docs in CD. Use Sphinx only if API cross-refs become the product. Keep Jupyter Book as an opt-in section for research-shaped projects/*.

🎨 Diagrams as code (inside the portal)

Architecture that only exists in slide decks drifts. Prefer text-native diagram tools in portal + leaf docs:

  • 🧜‍♂️ Mermaid [10] — flowcharts, C4-ish graphs, sequences; renders in GitHub/GitLab markdown and MkDocs plugins.
  • ✏️ D2 [11] — declarative layouts with stronger aesthetic defaults for system diagrams.
  • 🏛️ Structurizr / C4 [12] — when you need formal C4 model → views.

Minimum bar: every new services/* leaf ships a Mermaid (or D2) context diagram in its docs/ before the first deploy template lands.

🗺️ Rollout sequence

  1. 🧾 Freeze the directory contracts + import DAG in an ADR.
  2. 🧱 Stand up root uv workspace + shared ruff/mypy/poe/tox config.
  3. 🍪 Land cookiecutters/ + poe new-* tasks so new leaves are never hand-rolled.
  4. 👤 Turn on CODEOWNERS + branch protection / MR approval rules before the repo has many writers.
  5. 📦 Extract 2–3 already-shared modules into libs/ using hyphenated org namespaces (acme-platform-logging → acme/platform/logging).
  6. 📚 Stand up the docs/ portal landing page + poe docs CD publish; require a docs/ node on each new leaf.
  7. 🤖 Add .cursor/rules / Claude rules + a PR/MR review agent check wired to CONTRIBUTING.md.
  8. 📏 Require pytest on every leaf; tox matrices on shared libs; schedule a nightly coverage crawl for the whole monorepo.
  9. 🛰️ Migrate one microservice into services/ with path deps + a template pipeline.
  10. 🧬 Add one domain projects/ leaf that consumes the same libs.
  11. ☁️ Land baseline iac/modules + one stack wired to that service.
  12. 📈 Only then mass-migrate — with ownership, scaffolds, and affected-test CI already green.

🪙 My 2 cents

The monorepo pays rent only if the graph is real. Folder names alone are cosplay.

Make these contracts non-optional:

  • 🔗 Path dependencies + an enforceable import DAG
  • 🧩 Leaf CI child nodes under shared parent templates
  • 🏷️ Org-namespaced libs (foo-bar → foo/bar)
  • 🍪 poe new-* scaffolds (Copier / Cookiecutter / Projen) so teams never hand-roll leaves
  • 📚 First-class docs/ portal aggregated from every leaf
  • 🛡️ CODEOWNERS + ACL — unapproved path changes do not ship
  • 🐳 Shared dockers/ bases, not snowflake Dockerfiles
  • 💓 Repo-Health — pytest, tox matrices, nightly coverage crawl, symptom-based alerts to CODEOWNERS

Tooling posture:

  • ⚡ Start with uv + poe + forge-native CI
  • 🔥 Earn Blaze/Bazel or Buck/Buck2 later — don’t pay that tax on day one
  • 🤖 Encode the same contracts for AI (Cursor/Claude rules, MCP, PR/MR review agents)

🌱 Treat the repository itself as something you design — not something that accumulates.
